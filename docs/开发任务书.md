## 1. 后端开发文档

### 1.1 技术选型

| 类别 | 技术 | 版本 | 用途 |
|------|------|------|------|
| 包/环境管理 | **uv** | ≥ 0.6 | 虚拟环境创建 + 依赖安装（替代 pip + venv） |
| 框架 | FastAPI | ≥ 0.110 | REST API + WebSocket |
| 运行时 | Python | ≥ 3.11 | 异步支持 |
| ASGI 服务器 | Uvicorn | ≥ 0.27 | 生产运行 |
| HTTP 客户端 | httpx | ≥ 0.27 | Worker ↔ Master 通信（异步） |
| 数据验证 | Pydantic | ≥ 2.0 | 请求/响应模型 |
| 进程管理 | asyncio.subprocess | stdlib | Worker 执行任务 |
| 唯一 ID | uuid | stdlib | 任务 ID 生成 |

> **环境管理规范：** 后端统一使用 [uv](https://docs.astral.sh/uv/) 管理虚拟环境和依赖。禁止使用 `pip install` 或手动 `python -m venv`。
>
> ```bash
> # 初始化项目（在 backend/ 目录下）
> cd backend
> uv init --no-readme
> uv add fastapi uvicorn httpx pydantic
>
> # 后续添加依赖
> uv add <package>
>
> # 运行脚本
> uv run python run_master.py
> uv run python run_worker.py --port 8001 --cpu 4 --mem 8
> ```

### 1.2 项目结构

Master 采用企业级三层架构（Router / Service / Core），通过独立的 `schemas/` 模块严格规范所有 API 数据契约。Worker 保持轻量扁平结构。

```
backend/
├── master/
│   ├── __init__.py
│   ├── main.py                    # FastAPI 入口，挂载路由、注册生命周期事件
│   │
│   ├── core/                      # ── 基础设施层 ──
│   │   ├── __init__.py
│   │   ├── config.py              # 配置常量（心跳间隔、超时、端口、缓冲大小等）
│   │   ├── store.py               # 内存存储（全局 workers/tasks/logs 字典 + RingBuffer）
│   │   ├── events.py              # FastAPI 生命周期事件（startup/shutdown）
│   │   └── dependencies.py        # FastAPI 依赖注入（获取 store、scheduler 实例等）
│   │
│   ├── schemas/                   # ── 数据契约层 ──
│   │   ├── __init__.py
│   │   ├── task.py                # TaskSubmitRequest, TaskInfo, TaskStatusReport, TaskLogReport
│   │   ├── worker.py              # WorkerRegisterRequest, WorkerHeartbeatRequest, WorkerInfo
│   │   ├── response.py            # ApiResponse[T] 通用响应包装
│   │   └── ws.py                  # WebSocket 帧：ClusterStateFrame, LogWsFrame 等
│   │
│   ├── routers/                   # ── 路由层（Thin Controller）──
│   │   ├── __init__.py
│   │   ├── tasks.py               # /api/tasks 路由（提交、查询）
│   │   ├── workers.py             # /api/workers 路由（查询 Worker 列表）
│   │   └── internal.py            # /internal/* Worker 通信路由（注册、心跳、状态上报、日志上报）
│   │
│   ├── services/                  # ── 业务逻辑层 ──
│   │   ├── __init__.py
│   │   ├── task_service.py        # 任务创建、查询、状态流转
│   │   ├── worker_service.py      # Worker 注册、心跳更新、状态查询
│   │   ├── scheduler.py           # Bin Packing 调度器（Best-Fit + asyncio.Lock）
│   │   └── heartbeat.py           # 心跳监控后台任务（定时扫描、OFFLINE 处理）
│   │
│   └── ws/                        # ── WebSocket 管理 ──
│       ├── __init__.py
│       ├── cluster.py             # /ws/cluster 连接池管理 + 集群状态广播
│       └── logs.py                # /ws/logs/{task_id} 订阅管理 + 日志流推送
│
├── worker/
│   ├── __init__.py
│   ├── main.py                    # Worker FastAPI 入口 + 启动参数解析
│   ├── config.py                  # Worker 配置（Master 地址、心跳间隔等）
│   ├── schemas.py                 # Worker 侧 Pydantic 模型（ExecuteRequest 等）
│   ├── executor.py                # 子进程执行器（asyncio.subprocess）
│   └── reporter.py                # 向 Master 上报（注册、心跳、日志、状态）
│
├── requirements.txt
├── run_master.py                  # 启动脚本：python run_master.py
└── run_worker.py                  # 启动脚本：python run_worker.py --port 8001 --cpu 4 --mem 8

### 1.3 模块详细说明

#### Master — Core 层（基础设施）

##### master/core/config.py
```python

##### master/core/store.py
```python
from typing import Dict
from collections import deque

# 全局内存存储
workers: Dict[str, WorkerInfo] = {}
tasks: Dict[str, TaskInfo] = {}
task_logs: Dict[str, deque] = {}  # task_id -> deque(maxlen=LOG_BUFFER_SIZE)

##### master/core/events.py
- 注册 FastAPI `lifespan` 生命周期
- **startup**: 启动心跳监控后台任务、集群状态广播后台任务
- **shutdown**: 关闭所有 WebSocket 连接，取消后台任务

##### master/core/dependencies.py
- 提供 FastAPI `Depends()` 依赖注入
- 例如：`get_task_service()`, `get_worker_service()`, `get_scheduler()`
- 使 Router 层与 Service 层解耦

#### Master — Schemas 层（数据契约）

`schemas/` 是所有 REST API 和 WebSocket 数据结构的唯一真相来源（Single Source of Truth）。

##### master/schemas/task.py
```python
from pydantic import BaseModel, Field
from typing import Optional, Literal

class TaskSubmitRequest(BaseModel):
    """POST /api/tasks 请求体"""
    command: str = Field(..., min_length=1, max_length=1024)
    cpu_required: int = Field(..., ge=1)
    mem_required: int = Field(..., ge=1)

class TaskInfo(BaseModel):
    """任务完整信息（REST 响应 + WS 内部共享）"""
    task_id: str
    command: str
    cpu_required: int
    mem_required: int
    status: Literal["pending", "running", "success", "failed"]
    worker_id: Optional[str] = None
    created_at: str
    started_at: Optional[str] = None
    finished_at: Optional[str] = None

class TaskStatusReport(BaseModel):
    """POST /internal/task/{id}/status 请求体"""
    worker_id: str
    status: Literal["running", "success", "failed"]

class TaskLogReport(BaseModel):
    """POST /internal/task/{id}/log 请求体"""
    worker_id: str
    line_no: int
    content: str
```

##### master/schemas/worker.py
```python
from pydantic import BaseModel, Field
from typing import Literal

class WorkerRegisterRequest(BaseModel):
    """POST /internal/register 请求体"""
    worker_id: str
    host: str
    port: int
    total_cpu: int = Field(..., ge=1)
    total_mem: int = Field(..., ge=1)

class WorkerHeartbeatRequest(BaseModel):
    """POST /internal/heartbeat 请求体"""
    worker_id: str

class WorkerInfo(BaseModel):
    """Worker 完整信息（REST 响应 + WS 广播共享）"""
    worker_id: str
    host: str
    port: int
    total_cpu: int
    total_mem: int
    used_cpu: int = 0
    used_mem: int = 0
    status: Literal["online", "offline"] = "online"
    task_count: int = 0
    last_heartbeat: str
    registered_at: str
```

##### master/schemas/response.py
```python
from pydantic import BaseModel
from typing import TypeVar, Generic, Optional

T = TypeVar("T")

class ApiResponse(BaseModel, Generic[T]):
    """通用 REST API 响应包装"""
    code: int = 0
    message: str = "success"
    data: Optional[T] = None
```

##### master/schemas/ws.py
- `ClusterStateFrame`: 集群状态帧（type="cluster_state"）
- `LogConnectedFrame`: 日志连接确认帧（type="connected"）
- `LogHistoryFrame`: 历史日志帧（type="history"）
- `LogLineFrame`: 实时日志帧（type="log"）
- `TaskCompletedFrame`: 任务完成帧（type="task_completed"）

所有 WebSocket 帧结构均在此定义，确保 `ws/cluster.py` 和 `ws/logs.py` 使用统一的序列化格式。

#### Master — Router 层（Thin Controller）

Router 层只做三件事：**接收请求 → 调用 Service → 返回响应**。不包含任何业务逻辑。

##### master/routers/tasks.py
```python
# 路由层示例（伪代码）
@router.post("/api/tasks", response_model=ApiResponse[TaskInfo])
async def submit_task(
    req: TaskSubmitRequest,
    task_service: TaskService = Depends(get_task_service)
):
    task = await task_service.create_and_schedule(req)
    return ApiResponse(data=task)
```

- `POST /api/tasks` → 调用 `task_service.create_and_schedule()`
- `GET /api/tasks` → 调用 `task_service.list_tasks()`
- `GET /api/tasks/{task_id}` → 调用 `task_service.get_task()`

##### master/routers/workers.py
- `GET /api/workers` → 调用 `worker_service.list_workers()`

##### master/routers/internal.py
- `POST /internal/register` → 调用 `worker_service.register()`
- `POST /internal/heartbeat` → 调用 `worker_service.heartbeat()`
- `POST /internal/task/{task_id}/status` → 调用 `task_service.update_status()`
- `POST /internal/task/{task_id}/log` → 调用 `task_service.append_log()`

---

#### Master — Service 层（业务逻辑）

##### master/services/task_service.py
- `create_and_schedule(req)`: 创建任务 → 存入 store → 调用 scheduler 调度
- `list_tasks(status_filter)`: 查询任务列表，支持按状态过滤
- `get_task(task_id)`: 查询单个任务
- `update_status(task_id, report)`: 更新任务状态 → 终态时释放资源 → 触发重调度
- `append_log(task_id, report)`: 追加日志到 RingBuffer → 广播到 WebSocket 订阅者

##### master/services/worker_service.py
- `register(req)`: Worker 注册（幂等），初始化资源信息
- `heartbeat(req)`: 更新 `last_heartbeat` 时间戳
- `list_workers()`: 查询所有 Worker 列表
- `mark_offline(worker_id)`: 标记下线 → 其 Running 任务标记 Failed → 释放资源

##### master/services/scheduler.py
- 实现 Best-Fit Bin Packing 算法
- 使用 `asyncio.Lock` 保证调度原子性
- 核心函数：
  - `schedule_task(task) -> Optional[WorkerInfo]`
  - `try_reschedule_pending()`
  - `release_resources(task_id)`

##### master/services/heartbeat.py
- 后台 asyncio 定时任务
- 每 5 秒扫描所有 Worker 的 `last_heartbeat`
- 超时 Worker → 调用 `worker_service.mark_offline()` → 触发资源释放与重调度

---

#### Master — WebSocket 管理

##### master/ws/cluster.py
- 维护 `Set[WebSocket]` 连接池
- 后台每秒构建 `ClusterStateFrame` → 广播
- 使用 `schemas/ws.py` 中定义的帧结构进行序列化

##### master/ws/logs.py
- 维护 `Dict[task_id, Set[WebSocket]]` 订阅关系
- 连接时发送 `LogConnectedFrame` + `LogHistoryFrame`
- 新日志到达时广播 `LogLineFrame`
- 任务完成时发送 `TaskCompletedFrame`

- `report_log(task_id, line_no, content)`: POST /internal/task/{id}/log

#### api/index.ts
```typescript
import axios from 'axios'
import type { ApiResponse, TaskInfo } from '@/types'

const api = axios.create({

// 提交任务

  api.get<ApiResponse<{ workers: WorkerInfo[]; total: number }>>('/api/workers')
```

#### composables/useClusterWs.ts

**职责：** 维护到 `/ws/cluster` 的 WebSocket 连接，提供响应式的 `workers` 和 `tasksSummary`。

**核心逻辑：**
- 连接 `ws://master:8000/ws/cluster`
- 接收 `cluster_state` 帧 → 更新 `ref<WorkerInfo[]>` 和 `ref<TasksSummary>`
- 自动重连（指数退避，最多 5 次）
- 每 30 秒发送 `ping` 心跳
- 暴露 `workers`, `tasksSummary`, `wsStatus`

#### composables/useLogWs.ts

**职责：** 按需连接 `/ws/logs/{task_id}`，管理日志数组。

**核心逻辑：**
- `connect(taskId)` / `disconnect()`
- 接收 `history` 帧 → 填充已有日志
- 接收 `log` 帧 → 追加到日志数组
- 接收 `task_completed` 帧 → 标记结束
- 日志数组最多保留 5000 行（防止 DOM 过多）
- 暴露 `logs`, `isConnected`, `isCompleted`

#### composables/useAutoScroll.ts

**职责：** 管理日志区域的自动滚动行为。

**核心逻辑：**
- 监听日志数组变化 → `nextTick(() => container.scrollTop = container.scrollHeight)`
- 用户手动向上滚动 → 暂停自动滚动
- 用户滚回底部（距底部 < 50px）→ 恢复自动滚动
- 暴露 `containerRef`, `isAutoScrolling`, `scrollToBottom()`

## 4. 开发注意事项

### 4.1 后端注意事项

1. **CORS 配置：** Master main.py 必须添加 CORS 中间件，允许前端跨域访问
   ```python
   from fastapi.middleware.cors import CORSMiddleware
   app.add_middleware(CORSMiddleware, allow_origins=["*"], allow_methods=["*"], allow_headers=["*"])
   ```

2. **三层架构纪律：** Router 层只做请求解析和响应返回，禁止包含业务逻辑；Service 层不感知 HTTP 细节；Core 层不含业务逻辑

3. **Schema 唯一真相源：** 所有请求体/响应体的 Pydantic 模型统一在 `master/schemas/` 下定义，Router 和 Service 均从此处 import，严禁在其他模块重复定义

4. **调度器锁：** `scheduler.schedule_task()` 必须在 `asyncio.Lock` 内执行，防止并发超分

5. **日志上报性能：** Worker 每行日志一个 HTTP 请求，高频场景下可能成为瓶颈。MVP 阶段可接受，后续可优化为批量上报

6. **Worker 启动参数：** 通过命令行参数指定 `--port`, `--cpu`, `--mem`, `--master-url`，便于模拟多 Worker

7. **优雅关闭：** 通过 `core/events.py` 统一管理，Master 关闭时应关闭所有 WebSocket 连接、取消后台任务；Worker 关闭时应通知 Master（可选）

### 4.2 前端注意事项

1. **JSON snake_case 处理：** 后端返回 snake_case，前端内部用 camelCase。建议在 api 层做转换，或直接在类型定义中用 snake_case（推荐后者，减少复杂度）

2. **WebSocket 重连：** 使用指数退避，基准 1s，最大 5 次。用户可见的连接状态指示器

3. **日志防卡死：** 
   - 日志数组超过 5000 行时移除头部
   - `nextTick` + `requestAnimationFrame` 节流滚动
   - 避免在日志行中使用复杂的 Vue 模板语法

4. **响应式布局：** Worker 卡片使用 CSS Grid 或 Flex Wrap，适配不同数量的 Worker

5. **Element Plus 按需导入：** 使用自动导入插件减少打包体积

### 4.3 联调技巧

1. **模拟多 Worker：** 同一机器不同端口启动多个 Worker
   ```bash
   uv run python run_worker.py --port 8001 --cpu 4 --mem 8 --worker-id worker-01
   uv run python run_worker.py --port 8002 --cpu 8 --mem 16 --worker-id worker-02
   ```

2. **模拟长任务：** `echo "start" && sleep 10 && echo "done"`

3. **模拟大量日志：** `for i in $(seq 1 1000); do echo "line $i"; sleep 0.01; done`

4. **模拟 Worker 崩溃：** 直接 Ctrl+C 杀 Worker 进程，观察 Master 15 秒后标记 OFFLINE

5. **Bin Packing 验证命令：**
   ```bash
   # Worker A: 4核/8G
   uv run python run_worker.py --port 8001 --cpu 4 --mem 8

   # 提交两个 2核/4G 任务 → 应该都分配到 Worker A
   curl -X POST http://localhost:8000/api/tasks -H "Content-Type: application/json" \
     -d '{"command": "sleep 30", "cpu_required": 2, "mem_required": 4}'
   curl -X POST http://localhost:8000/api/tasks -H "Content-Type: application/json" \
     -d '{"command": "sleep 30", "cpu_required": 2, "mem_required": 4}'

   # 查看两个任务的 worker_id 应该相同
   curl http://localhost:8000/api/tasks
   ```
