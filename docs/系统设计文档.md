## 1. 系统总体架构

```
┌─────────────────────────────────────────────────────┐
│                     Browser (Vue 3)                  │
│  ┌──────────┐  ┌──────────────┐  ┌───────────────┐  │
│  │ 集群热力图 │  │  任务管理面板  │  │  日志流模态框  │  │
│  └─────┬────┘  └──────┬───────┘  └───────┬───────┘  │
│        │ WS           │ HTTP             │ WS        │
└────────┼──────────────┼─────────────────┼───────────┘
         │              │                 │
    ─────┼──────────────┼─────────────────┼──── Network
         │              │                 │
┌────────┴──────────────┴─────────────────┴───────────┐
│                   Master (FastAPI)                    │
│  ┌──────────┐  ┌──────────┐  ┌───────────────────┐  │
│  │ REST API  │  │ WS Hub   │  │  Scheduler        │  │
│  │ Router    │  │ Manager  │  │  (Bin Packing)    │  │
│  └────┬─────┘  └────┬─────┘  └────────┬──────────┘  │
│       │              │                 │              │
│  ┌────┴──────────────┴─────────────────┴──────────┐  │
│  │              State Store (In-Memory)            │  │
│  │  workers: Dict[worker_id, WorkerState]          │  │
│  │  tasks:   Dict[task_id, TaskState]              │  │

│  └────────────────────────────────────────────────┘  │
└──────────────┬────────────────────────┬──────────────┘
               │ HTTP                   │ HTTP
         ┌─────┴─────┐           ┌─────┴─────┐
         │  Worker 1  │           │  Worker N  │
         │ (FastAPI)  │           │ (FastAPI)  │
         └───────────┘           └───────────┘
```

### 1.1 架构风格

**单 Master + 多 Worker**，Master 和 Worker 均为独立进程。

- **通信协议：** 全部基于 HTTP/WebSocket（而非 RPC/gRPC），保持轻量。
- **状态存储：** 纯内存（Dict），无数据库依赖。MVP 阶段重启即丢失数据。
- **进程模型：** Master 一个 FastAPI 进程；每个 Worker 一个独立 FastAPI 进程（可在同一机器用不同端口模拟）。

### 1.2 核心设计决策

| 决策 | 选择 | 理由 |
|------|------|------|
| Worker → Master 通信 | HTTP（注册、心跳、日志上报、任务状态回调） | 无状态、实现简单、天然可重试 |
| Master → 前端（集群状态） | WebSocket 推送 | 需要实时更新，轮询延迟高且浪费带宽 |
| Master → 前端（任务日志） | WebSocket 推送（按 task_id 订阅） | 低延迟流式日志 |
| 调度策略 | Best-Fit Decreasing（二维 Bin Packing） | 资源利用率最大化 |
| 数据存储 | 内存 Dict | 轻量级，无外部依赖 |
| Worker 进程 | 独立 FastAPI 服务 | Worker 需要接收 Master 下发的任务（HTTP API） |

## 2. Master 职责

Master 是系统的"大脑"，承担以下职责：

### 2.1 REST API 服务

| API | 方法 | 说明 |
|-----|------|------|
| `/api/tasks` | POST | 接收任务提交 |
| `/api/tasks` | GET | 查询任务列表 |
| `/api/tasks/{task_id}` | GET | 查询单个任务详情 |
| `/api/workers` | GET | 查询 Worker 列表 |

### 2.2 WebSocket 服务

| 端点 | 说明 |
|------|------|
| `/ws/cluster` | 集群状态广播（Worker 列表 + 资源占用，每秒推送） |
| `/ws/logs/{task_id}` | 任务日志流推送（订阅特定任务的 stdout） |

### 2.3 Worker 管理 API（供 Worker 调用）

| API | 方法 | 说明 |
|-----|------|------|
| `/internal/register` | POST | Worker 注册 |
| `/internal/heartbeat` | POST | Worker 心跳上报 |
| `/internal/task/{task_id}/status` | POST | Worker 回报任务状态变更 |
| `/internal/task/{task_id}/log` | POST | Worker 上报日志行 |

### 2.4 调度器（Scheduler）

- 接收到任务提交请求后，执行 Bin Packing 算法选择 Worker
- 调度成功 → 通过 HTTP 调用 Worker 的执行接口
- 调度失败（无可用资源）→ 任务保持 Pending，等待资源释放后重试

### 2.5 心跳监控器（Heartbeat Monitor）

- 后台定时任务（asyncio），每 5 秒扫描一次
- Worker 超过 15 秒未心跳 → 标记 OFFLINE
- OFFLINE Worker 上的 Running 任务 → 标记 Failed，释放虚拟资源
- 资源释放后 → 触发 Pending 任务重新调度

## 3. Worker 职责

Worker 是系统的"手脚"，职责单一明确：

### 3.1 生命周期

```
Worker 启动
    │
    ▼
POST /internal/register (向 Master 注册，上报 CPU 核数 + 内存大小)
    │
    ▼
定时循环 (每 5 秒)
    │── POST /internal/heartbeat (心跳上报)
    │
    ▼
[空闲等待 Master 下发任务]
    │
    ▼
收到 POST /execute (Master 调用)
    │
    ├── 启动子进程执行 command
    ├── POST /internal/task/{id}/status → Running
    ├── 逐行读取 stdout → POST /internal/task/{id}/log
    ├── 子进程结束 → POST /internal/task/{id}/status → Success/Failed
    │
    ▼
[回到空闲等待]
```

### 3.2 Worker API（供 Master 调用）

| API | 方法 | 说明 |
|-----|------|------|
| `/execute` | POST | 接收并执行任务（command、task_id、cpu_required、mem_required） |

### 3.3 子进程管理

- 使用 `asyncio.create_subprocess_exec` 启动任务命令
- 通过 `stdout=asyncio.subprocess.PIPE` 捕获输出
- 逐行读取 stdout（`readline()`），每行即时上报 Master
- 进程退出码 0 → Success，非 0 → Failed

## 5. 任务生命周期

```
            任务提交 (REST API)
                 │
                 ▼
            ┌─────────┐
            │ Pending  │ ← 等待调度（资源不足时停留在此）
            └────┬────┘
                 │ Scheduler 选中 Worker
                 ▼
            ┌─────────┐
            │ Running  │ ← Worker 正在执行
            └────┬────┘
                 │
          ┌──────┴──────┐
          ▼              ▼
    ┌─────────┐    ┌─────────┐
    │ Success  │    │ Failed  │
    └─────────┘    └─────────┘
```

### 5.1 状态转换规则

| 当前状态 | 事件 | 目标状态 |
|----------|------|----------|
| (新建) | 任务提交到 Master | Pending |
| Pending | Scheduler 分配到 Worker 且 Worker 确认开始执行 | Running |
| Running | 子进程退出码 == 0 | Success |
| Running | 子进程退出码 != 0 | Failed |
| Running | Worker 心跳超时（OFFLINE） | Failed |

### 5.2 状态转换约束

- **单向流动**：状态只能向右移动，不可回退（Failed 不能变回 Running）
- **终态不可变**：Success 和 Failed 是终态，一旦到达不再变化
- **Pending → Running 需要原子性**：调度器分配 Worker 时必须在锁内完成资源扣减

## 6. Worker 生命周期

```
            Worker 启动
                │
                ▼
           ┌────────┐
           │ ONLINE  │ ← 已注册，心跳正常
           └────┬───┘
                │ 心跳超时 (>15s)
                ▼
           ┌─────────┐
           │ OFFLINE  │ ← 心跳丢失
           └────┬────┘
                │ 重新收到心跳
                ▼
           ┌────────┐
           │ ONLINE  │ ← 恢复在线
           └────────┘
```

### 6.1 Worker 状态定义

| 状态 | 条件 |
|------|------|
| ONLINE | 已注册且最近 15 秒内有心跳 |
| OFFLINE | 超过 15 秒未收到心跳 |

### 6.2 OFFLINE 处理流程

1. 心跳监控器检测到超时 → 标记 Worker 为 OFFLINE
2. 查找该 Worker 上所有 Running 任务 → 标记为 Failed
3. 释放这些任务占用的虚拟 CPU/内存资源
4. 尝试重新调度 Pending 任务（新释放的资源可能满足）
5. WebSocket 推送更新的集群状态 → 前端卡片变灰

## 7. 调度策略：Bin Packing 详细设计

### 7.1 算法选择：Best-Fit

选择 **Best-Fit** 策略而非 First-Fit，原因：
- Best-Fit 倾向于将任务放到"刚好够用"的 Worker，减少资源碎片
- 对于小规模集群（< 100 Worker），Best-Fit 的 O(N) 遍历性能完全可接受

### 7.2 二维资源评分

由于需要同时考虑 CPU 和内存，采用**归一化加权评分**：

```python
def score_worker(worker, task):
    """分数越低越好（剩余资源越少 = 越紧凑）"""
    if worker.remaining_cpu < task.cpu_required:
        return None  # 不满足
    if worker.remaining_mem < task.mem_required:
        return None  # 不满足
    
    cpu_ratio = (worker.remaining_cpu - task.cpu_required) / worker.total_cpu
    mem_ratio = (worker.remaining_mem - task.mem_required) / worker.total_mem
    
    return cpu_ratio + mem_ratio  # 综合剩余比例，越小越紧凑
```

### 7.3 调度流程

```python
async def schedule_task(task):
    async with scheduler_lock:  # 互斥锁防止并发超分
        best_worker = None
        best_score = float('inf')
        
        for worker in workers.values():
            if worker.status != "ONLINE":
                continue
            score = score_worker(worker, task)
            if score is not None and score < best_score:
                best_score = score
                best_worker = worker
        
        if best_worker is None:
            task.status = "Pending"  # 无可用 Worker，等待
            return False
        
        # 原子性扣减资源
        best_worker.remaining_cpu -= task.cpu_required
        best_worker.remaining_mem -= task.mem_required
        task.worker_id = best_worker.id
        task.status = "Running"
        
        # 异步通知 Worker 执行（不在锁内等待）
        asyncio.create_task(dispatch_to_worker(best_worker, task))
        return True
```

### 7.4 文档测试用例验证

**场景：** Worker A 有 4核/8G，提交两个各需 2核/4G 的任务。

1. Task 1 提交 → schedule_task 执行
   - Worker A: remaining_cpu=4, remaining_mem=8
   - score = (4-2)/4 + (8-4)/8 = 0.5 + 0.5 = 1.0 ✓
   - 分配后: remaining_cpu=2, remaining_mem=4

2. Task 2 提交 → schedule_task 执行
   - Worker A: remaining_cpu=2, remaining_mem=4
   - score = (2-2)/4 + (4-4)/8 = 0.0 + 0.0 = 0.0 ✓（刚好满足）
   - 分配后: remaining_cpu=0, remaining_mem=0

3. Task 3 提交（假设也需 2核/4G）→ Worker A 不满足 → 保持 Pending ✓

### 7.5 Pending 任务重调度

**触发时机：**
- 任务完成（Success/Failed）→ 资源归还后
- Worker OFFLINE → 该 Worker 上任务标记 Failed → 资源释放后
- 新 Worker 注册

**流程：**
```python
async def try_reschedule_pending():
    """尝试调度所有 Pending 任务"""
    pending_tasks = [t for t in tasks.values() if t.status == "Pending"]
    for task in pending_tasks:
        await schedule_task(task)
```

## 8. 实时通信方案

### 8.1 通信链路总览

```
Worker ──HTTP POST──> Master ──WebSocket──> Browser
(心跳/日志/状态)      (存储+广播)         (渲染)
```

### 8.2 集群状态推送

| 项目 | 设计 |
|------|------|
| 协议 | WebSocket (`/ws/cluster`) |
| 推送频率 | 每 1 秒推送一次全量 Worker 列表 |
| 数据格式 | JSON（Worker 列表含资源使用情况） |
| 触发方式 | Master 后台定时任务广播给所有连接的客户端 |
| 连接管理 | Master 维护 `Set[WebSocket]`，客户端断开自动移除 |

### 8.3 任务日志推送

| 项目 | 设计 |
|------|------|
| 协议 | WebSocket (`/ws/logs/{task_id}`) |
| 推送频率 | 即时（Worker 上报一行，Master 立刻推送） |
| 数据格式 | JSON（包含 task_id、行号、内容、时间戳） |
| 订阅模型 | 前端连接时指定 task_id → Master 维护 `Dict[task_id, Set[WebSocket]]` |
| 历史日志 | 连接时先推送缓存的历史日志（RingBuffer 最近 1000 行） |
| 连接关闭 | 前端关闭模态框 → 断开 WebSocket → Master 清理订阅 |

### 8.4 日志数据流详细路径

```
Worker 子进程
    │ stdout (逐行)
    ▼
Worker asyncio 读取循环
    │ readline()
    ▼
HTTP POST /internal/task/{task_id}/log
    │ body: { "task_id": "xxx", "line": "...", "line_no": 42 }
    ▼
Master 接收
    │
    ├── 存入 RingBuffer (logs[task_id])
    │
    └── 广播给所有订阅该 task_id 的 WebSocket 客户端
              │
              ▼
         Browser 渲染日志行 + 自动滚动
```

### 8.5 日志缓冲与防卡死策略

| 层 | 策略 |
|----|------|
| Worker | 逐行读取（readline），不做缓冲（保证实时性） |
| Master | RingBuffer 保留最近 1000 行/task，防止内存溢出 |
| 前端 | ① 日志行合并（50ms 节流，batch 渲染）<br>② 最多保留 5000 行 DOM 节点，超出后移除头部<br>③ 用户手动上滚时暂停自动滚动，滚回底部时恢复 |

## 9. 后端项目结构

Master 采用企业级三层架构（Router / Service / Core），通过独立的 `schemas/` 模块严格规范所有 REST API 和 WebSocket 的请求体与响应体字段。Worker 保持轻量扁平结构。

```
backend/
├── master/
│   ├── __init__.py
│   ├── main.py                    # FastAPI 入口，挂载路由、注册生命周期事件
│   │
│   ├── core/                      # ── 基础设施层 ──
│   │   ├── __init__.py
│   │   ├── config.py              # 配置常量（心跳间隔、超时、端口、缓冲大小等）
│   │   ├── store.py               # 内存存储（全局 workers/tasks/logs 字典 + RingBuffer）
│   │   ├── events.py              # FastAPI 生命周期事件（startup/shutdown）
│   │   └── dependencies.py        # FastAPI 依赖注入（获取 store、scheduler 实例等）
│   │
│   ├── schemas/                   # ── 数据契约层 ──
│   │   ├── __init__.py
│   │   ├── task.py                # 任务相关：TaskSubmitRequest, TaskInfo, TaskStatusReport, TaskLogReport
│   │   ├── worker.py              # Worker 相关：WorkerRegisterRequest, WorkerHeartbeatRequest, WorkerInfo
│   │   ├── response.py            # 通用响应包装：ApiResponse[T]
│   │   └── ws.py                  # WebSocket 帧：ClusterStateFrame, LogWsFrame 等
│   │
│   ├── routers/                   # ── 路由层（Thin Controller）──
│   │   ├── __init__.py
│   │   ├── tasks.py               # /api/tasks 路由（提交、查询）
│   │   ├── workers.py             # /api/workers 路由（查询 Worker 列表）
│   │   └── internal.py            # /internal/* Worker 通信路由（注册、心跳、状态上报、日志上报）
│   │
│   ├── services/                  # ── 业务逻辑层 ──
│   │   ├── __init__.py
│   │   ├── task_service.py        # 任务创建、查询、状态流转
│   │   ├── worker_service.py      # Worker 注册、心跳更新、状态查询
│   │   ├── scheduler.py           # Bin Packing 调度器（Best-Fit + asyncio.Lock）
│   │   └── heartbeat.py           # 心跳监控后台任务（定时扫描、OFFLINE 处理）
│   │
│   └── ws/                        # ── WebSocket 管理 ──
│       ├── __init__.py
│       ├── cluster.py             # /ws/cluster 连接池管理 + 集群状态广播
│       └── logs.py                # /ws/logs/{task_id} 订阅管理 + 日志流推送
│
├── worker/
│   ├── __init__.py
│   ├── main.py                    # Worker FastAPI 入口 + 启动参数解析
│   ├── config.py                  # Worker 配置（Master 地址、心跳间隔等）
│   ├── schemas.py                 # Worker 侧 Pydantic 模型（ExecuteRequest 等）
│   ├── executor.py                # 子进程执行器（asyncio.subprocess）
│   └── reporter.py                # 向 Master 上报（注册、心跳、日志、状态）
│
├── requirements.txt
├── run_master.py                  # 启动脚本：python run_master.py
└── run_worker.py                  # 启动脚本：python run_worker.py --port 8001 --cpu 4 --mem 8
```

### 9.1 Master 分层职责

| 层 | 目录 | 职责 | 禁止 |
|----|------|------|------|
| **Router** | `routers/` | 接收 HTTP 请求，参数解析（由 schemas 验证），调用 Service，返回响应 | 不含业务逻辑、不直接操作 store |
| **Service** | `services/` | 业务逻辑实现（调度、心跳检测、状态流转、资源管理） | 不感知 HTTP 细节、不直接构造 Response |
| **Core** | `core/` | 基础设施：配置、内存存储、生命周期事件、依赖注入 | 不含业务逻辑 |
| **Schema** | `schemas/` | Pydantic 模型定义：请求体、响应体、WS 帧结构 | 不含逻辑，纯数据结构 |
| **WebSocket** | `ws/` | WS 连接池管理与消息广播 | 不含调度/心跳等业务逻辑 |

### 9.2 schemas/ 模块设计

`schemas/` 是所有 REST API 和 WebSocket 数据结构的唯一真相来源（Single Source of Truth），确保前后端字段一致性。

| 文件 | 包含的模型 | 用途 |
|------|-----------|------|
| `task.py` | `TaskSubmitRequest`, `TaskInfo`, `TaskStatusReport`, `TaskLogReport` | 任务相关请求体与数据模型 |
| `worker.py` | `WorkerRegisterRequest`, `WorkerHeartbeatRequest`, `WorkerInfo` | Worker 相关请求体与数据模型 |
| `response.py` | `ApiResponse[T]` | 通用响应包装（code/message/data） |
| `ws.py` | `ClusterStateFrame`, `LogConnectedFrame`, `LogHistoryFrame`, `LogLineFrame`, `TaskCompletedFrame` | WebSocket 帧结构定义 |

```
frontend/
├── public/
├── src/
│   ├── api/
```
