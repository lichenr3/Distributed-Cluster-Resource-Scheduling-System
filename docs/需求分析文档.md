> 项目：轻量级分布式任务调度系统 (Mini-Scheduler)

> 日期：2026-03-13  
> 来源：睿思芯科AI挑战题(1).pdf

---

## 1. 项目概述

为计算集群构建一个**可视化的轻量级分布式任务调度系统**。系统采用 Master/Worker 架构，支持任务提交、智能调度（Bin Packing）、实时状态追踪和日志流推送。要求后台稳定、前端炫酷且实时。

**技术栈约束（用户指定）：**
- 后端：Python + FastAPI
- 前端：Vue 3 (Composition API)

## 3. 难点分析

### 3.1 Bin Packing 调度算法（核心难点）

**需求原文场景：**
> Worker A 有 4核/8G 资源，提交两个各需 2核/4G 的任务 → 应能**同时**调度至该 Worker。

**难点拆解：**

| 难点 | 说明 | 应对策略 |
|------|------|----------|
| 二维资源匹配 | 同时考虑 CPU 和内存两个维度，不是简单的一维装箱 | 采用 Best-Fit Decreasing 策略：对每个 Worker 计算综合剩余资源得分，选择"最紧凑"的 Worker |
| 并发资源扣减 | 同一 Worker 可同时运行多个任务，剩余资源需动态维护 | Master 内存中维护每个 Worker 的 `remaining_cpu` 和 `remaining_mem`，任务分配时即扣减，任务完成时归还 |
| 无可用 Worker | 所有 Worker 剩余资源不足 | 任务保持 `Pending` 状态，后续通过定时重调度或资源释放事件触发再次尝试 |
| Worker 中途下线 | 任务已分配但 Worker 挂掉 | 将该 Worker 上 Running 状态的任务标记为 `Failed`，归还虚拟资源 |

### 3.2 实时日志推流（核心难点）

**数据流路径：** `Worker 子进程 stdout` → `Worker 进程` → `Master (WebSocket/HTTP)` → `前端 (WebSocket)`

**难点拆解：**

| 难点 | 说明 | 应对策略 |
|------|------|----------|
| Worker → Master 传输 | Worker 需实时将 subprocess stdout 推给 Master | Worker 使用 `asyncio.create_subprocess_exec`，逐行读取 stdout，通过 HTTP POST 或 WebSocket 推送给 Master |
| Master → 前端推送 | 前端需实时接收日志 | Master 维护 WebSocket 连接，按 task_id 订阅，有新日志时推送 |
| 日志量大导致卡死 | 高频日志冲击前端 DOM | 前端采用：① 节流（throttle）合并日志行 ② 虚拟列表限制渲染行数 ③ 仅保留最近 N 行 |
| 自动滚动到底部 | 新日志到来时自动滚动 | `scrollTop = scrollHeight`，但用户手动上滚时应暂停自动滚动，滚到底部时恢复 |
| 断线重连 | WebSocket 连接中断 | 前端实现指数退避重连，重连后请求缺失日志（Master 缓存最近日志） |

### 3.3 Worker 心跳与 OFFLINE 检测

| 难点 | 说明 | 应对策略 |
|------|------|----------|
| 心跳协议选择 | HTTP 轮询 vs WebSocket 长连接 | **推荐 HTTP 轮询心跳**：Worker 每 N 秒 POST 心跳至 Master，实现简单、无状态、容错好 |
| 超时判定 | 多久没心跳算 OFFLINE | 心跳间隔 5s，超时阈值 15s（3 次未收到心跳） |
| OFFLINE 后任务处理 | Worker 下线后其上运行的任务怎么办 | 标记为 `Failed`，释放虚拟资源，Pending 任务可被重新调度 |

## 4. 隐含需求（原文未显式提及但必须实现）

| 编号 | 隐含需求 | 推理依据 |
|------|----------|----------|
| H-1 | Worker 心跳机制 | 前端要显示 OFFLINE 状态 → Master 必须检测 Worker 存活 |
| H-2 | Worker → Master 日志上报通道 | 前端要显示实时日志 → 日志必须从 Worker 传到 Master |
| H-3 | 任务完成后资源归还 | Bin Packing 基于"剩余资源" → 任务结束后必须归还 CPU/内存 |
| H-4 | 任务列表查询接口 | 前端需展示任务 → 必须有获取任务列表的 API |
| H-5 | Worker 列表查询接口 | 前端热力图需展示所有 Worker → 必须有获取 Worker 列表的 API |
| H-6 | Pending 任务重调度 | 当 Worker 资源释放时，Pending 任务应尝试重新调度 |
| H-7 | 任务提交表单 | 前端需要有地方提交任务（command、cpu、mem） |

## 5. 风险点

| 风险 | 影响 | 等级 | 缓解措施 |
|------|------|------|----------|
| Bin Packing 调度并发竞争 | 两个任务同时提交，可能超分 Worker 资源 | 高 | Master 调度器使用互斥锁（asyncio.Lock），保证串行调度 |
| WebSocket 连接数过多 | 多个前端同时查看日志，连接爆炸 | 中 | 限制每个 task_id 的最大订阅数；日志广播而非单播 |
| 大日志撑爆内存 | 长时间运行任务产生 GB 级日志 | 中 | Master 仅缓存最近 1000 行/task；Worker 端也做环形缓冲 |
| Worker 注册风暴 | 大量 Worker 同时启动注册 | 低 | FastAPI 异步天然扛得住；可加限流 |
| 前端性能 | 大量 Worker 卡片 + 实时更新 | 中 | 使用虚拟列表、requestAnimationFrame 节流更新频率 |
| 原文无评分标准 | 不确定各功能权重 | 中 | 优先保证核心功能完整，UI 美观为加分项 |

## 6. MVP（最小可交付版本）范围

### 6.1 MVP 必须包含

| 层 | 功能 | 说明 |
|----|------|------|
| 后端 | Master 进程 | REST API（任务提交、任务列表、Worker 列表）+ WebSocket（日志推送、集群状态推送）+ Bin Packing 调度器 |
| 后端 | Worker 进程 | 自动注册 + 心跳上报 + 任务执行（subprocess）+ stdout 日志上报 |
| 前端 | 集群热力图 | Worker 卡片 + CPU/内存进度条 + OFFLINE 状态 + WebSocket 实时更新 |
| 前端 | 任务管理 | 任务提交表单 + 任务列表 + 状态徽标 |
| 前端 | 日志流 | 模态框 + WebSocket 实时日志 + 自动滚动 |

### 6.2 MVP 不包含（可后续迭代）

| 功能 | 原因 |
|------|------|
| 用户认证/权限 | 原文未要求 |
| 任务优先级队列 | 原文未要求，Bin Packing 已满足 |
| 持久化存储（数据库） | 原文未要求，MVP 用内存存储即可 |
| 任务重试/重新调度到其他 Worker | 原文只要求标记 Failed，未要求自动重试 |
| Worker 动态扩缩容 | 原文未要求 |
| 日志持久化到文件/数据库 | MVP 用内存环形缓冲即可 |

### 6.3 假设清单

| 编号 | 假设 | 说明 |
|------|------|------|
| A-1 | Master 和 Worker 在同一局域网内通信 | 不考虑跨公网、TLS 等 |
| A-2 | Worker 数量 < 100 | 轻量级系统，无需考虑大规模集群 |
| A-3 | 任务执行时间 < 1 小时 | 不需要 checkpoint/恢复机制 |
| A-4 | 单 Master 架构（无高可用） | 原文未要求 Master 冗余 |
| A-5 | 内存存储足够 | 不使用数据库，重启后数据丢失可接受 |
| A-6 | Worker 的 CPU/内存为静态上报值 | 不需要实时采集实际使用率（调度基于"分配量"而非"实际用量"） |
| A-7 | 前端仅支持现代浏览器 | Chrome/Firefox/Edge 最新版 |

## 7. 术语表

| 术语 | 定义 |
|------|------|
| Master | 调度中心进程，接收任务、管理 Worker、执行调度算法 |
| Worker | 工作节点进程，注册到 Master，执行分配的任务 |
| Bin Packing | 装箱算法，将任务（物品）分配到 Worker（箱子）中，使资源利用最大化 |
| Heartbeat | 心跳，Worker 定期向 Master 发送存活信号 |
| OFFLINE | Worker 心跳超时后的状态 |
| Live Log Stream | 实时日志流，将任务 stdout 通过 WebSocket 推送到前端 |
